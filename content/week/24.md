### Working With Data

This is similar to what we did [with Are&period;na](/week/18/#the-are-na-api)—but now working with a JSON file *within* your project. You could use this as a model for an “API” of your own data/content:
<!-- .balance -->

[<cite>`local-json`</cite>](https://github.com/typography-interaction-2526/local-json)

- Remember [JSON](https://en.wikipedia.org/wiki/JSON) (*JavaScript Object Notation*)?

	- Before we asked Are&period;na for data/content, but you can also include it yourself “locally”
	- You can write/edit your own `.json` files by hand, though [the syntax](https://developer.mozilla.org/en-US/docs/Learn/JavaScript/Objects/JSON) is pretty easy to mess up!
	- Instead you could start your data from [Google Sheets](https://docs.google.com/spreadsheets/d/1Ce0ZXUy7ne3-XKs70dAHjAriu4PPMAkEBlgFjp6Q5mc/edit#gid=0), and use [an extension](https://chromewebstore.google.com/detail/sheets-to-json/enmkalgdnmcaljdfkojckdbhkjmffmoa) to export—or [download a `.csv`](https://docs.gravitykit.com/article/473-exporting-a-csv-from-google-sheets#:~:text=Exporting%20a%20CSV,%C2%B7%20Importing%20Your) and [convert it](https://smallcsv.com/csv-to-json)
	- There is also [a better add-on](https://workspace.google.com/marketplace/app/export_sheet_data/903838927001), but New School IT doesn’t want us to have nice things—use a personal account to try this out!

- Just like with our Are&period;na projects, we can generate DOM from this data!

	- IRL, you’d probably have some kind of templating framework or library—but you can use vanilla JS for repetitive DOM
	- We’ll again use a [`forEach` loop](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Array/forEach) to go through our [*array*](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Array) of data
	- We can use the same [template literals/strings](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Template_literals), in our JS file
	- Using [`if`/`else`](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Statements/if...else) *conditional logic* to apply a class
	- This still uses [the `fetch` API](https://developer.mozilla.org/en-US/docs/Web/API/Fetch_API/Using_Fetch) to get the data—but now from our local `.json` file!

### Keeping Track of State

And an example of saving [*state*](https://en.wikipedia.org/wiki/State_(computer_science)). This is a little more advanced, but it shows some ways you might retain a visitor’s preferences/selections within your project (within what our free GitHub hosting allows):
<!-- .balance -->

[<cite>`forms-params-storage`</cite>](https://github.com/typography-interaction-2526/forms-params-storage)

- Briefly, [*static* vs. *dynamic*](https://about.gitlab.com/blog/2016/06/03/ssg-overview-gitlab-pages-part-1-dynamic-x-static/) websites (in a build/hosting context)

	- We have been writing *static* sites—meaning that only you can change your pages, in code
	- [GitHub Pages](https://docs.github.com/en/pages/getting-started-with-github-pages/about-github-pages) (our hosting) does not allow server-side languages (PHP, Ruby, Go, Next.JS, Node, etc.) nor databases (My&NoBreak;SQL, Postgres, etc.)
	- These are how most sites are made *dynamic*—constructing pages, storing information, responding to visitor requests, allowing sign-in
	- But we can *approximate/suggest* some dynamic behavior with JS!
	<!-- .balance -->

- Then onto [`<form>`](https://developer.mozilla.org/en-US/docs/Web/HTML/Element/form), [`<input>`](https://developer.mozilla.org/en-US/docs/Web/HTML/Element/input), [`<select>`](https://developer.mozilla.org/en-US/docs/Web/HTML/Element/select), and [`<buttton>`](https://developer.mozilla.org/en-US/docs/Web/HTML/Element/button) elements

	Comment out the `form-state.js` to start! <!-- .note -->

	- `<form>` / `<input>` / `<select>` / `<button>` are semantic indicators of in-page interactivity
	- The [`type=` attribute](https://developer.mozilla.org/en-US/docs/Web/HTML/Reference/Elements/input#input_types) is used to show different types of controls, made by the browser—here with user-agent styles

	Comment in/out the `reset.css`&#x202F;!<!-- .note -->

	- The [`required` attribute](https://developer.mozilla.org/en-US/docs/Web/HTML/Reference/Attributes/required) makes them… required!
	- [`<label>`](https://developer.mozilla.org/en-US/docs/Web/HTML/Element/label) elements are connected to these with matching `for=`, `id=`, `name=` attributes
	- You can (and should) [style them](https://developer.mozilla.org/en-US/docs/Learn/Forms/Advanced_form_styling) from these defaults!
	<!-- .balance -->

- Onto [search](https://developer.mozilla.org/en-US/docs/Web/API/Location/search) (aka: *query strings*) and [URL params](https://developer.mozilla.org/en-US/docs/Web/API/URLSearchParams) (for *parameters*)

	- By default, forms add their *query string* to the URL (in JS, called the [`location`](https://developer.mozilla.org/en-US/docs/Web/API/Location)) on [the `submit` event](https://developer.mozilla.org/en-US/docs/Web/API/HTMLFormElement/submit)
	- This is a way for your request to the server to be more specific—what did the visitor input on this page?
	- In a URL the query follows the trailing `?`&#x202F;, each parameter is `name=value`&#x202F;, appended with an `&`&#x202F;:

	```
	https://example.com/?param=something&another-param=else
	```

	- We can use these to track our page’s *state*—what the visitor has done—via the changes in the URL
	- Using URL&NoBreak;s also makes these *shareable*—you could bookmark/text message/&NoBreak;QR&NoBreak;-code to different states!

	Comment `form-state.js` back in and go through! <!-- .note -->

	- [`preventDefault`](https://developer.mozilla.org/en-US/docs/Web/API/Event/preventDefault) stops the form from submitting (which would refresh the page)
	- We’ll instead use [the combined `FormData`](https://developer.mozilla.org/en-US/docs/Web/API/FormData) to construct our own parameter string
	- Then we’ll update the current URL on [the `input` event](https://developer.mozilla.org/en-US/docs/Web/API/Element/input_event) (meaning any change to the form)
	- The visitor’s changes and URL are then in sync!
	<!-- .balance -->

- Finally [web storage](https://developer.mozilla.org/en-US/docs/Web/API/Web_Storage_API/Using_the_Web_Storage_API), to keep our state!

	- This API [*persists*](https://en.wikipedia.org/wiki/Persistence_(computer_science)) data across reloads/sessions—but it is still limited to a single device/browser
	- [Cookies](https://developer.mozilla.org/en-US/docs/Web/HTTP/Guides/Cookies) (cross domain, visible to server) vs. [`sessionStorage`](https://developer.mozilla.org/en-US/docs/Web/API/Window/sessionStorage) (this session) vs. [`localStorage`](https://developer.mozilla.org/en-US/docs/Web/API/Window/localStorage) (this browser)—are all just ways to keep track of things
	- These are visible/editable in [a DevTools](/topic/dev-tools/#inspecting-pages) panel: <samp>Application</samp>&#x202F;→&thinsp;<samp>Storage</samp>&#x202F;→&thinsp;<samp>Local Storage</samp>
