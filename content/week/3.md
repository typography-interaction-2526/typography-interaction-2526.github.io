## Let’s Make a Web Page

Our whole second half today will be a demo, setting us up for next week’s assignment and our first projects:
<!-- .balance -->

1. You should have downloaded [Visual Studio Code](https://code.visualstudio.com), which we’ll demonstrate with as our text editor for the course.

	You’ll hear us refer to this as an [IDE](https://en.wikipedia.org/wiki/Integrated_development_environment), for *Integrated Development Environment*. Think of this as a fancy text editor with coding-specific features, like syntax highlighting and tabs. There are many of them! *VS Code* is probably the most popular, right now. (It is made by Microsoft.)

1. You also should have downloaded [GitHub Desktop](https://desktop.github.com), which we’ll use to talk to [GitHub](http://github.com) (the site) and manage our code. Let’s [sign into the app](https://docs.github.com/en/desktop/installing-and-authenticating-to-github-desktop/authenticating-to-github-in-github-desktop) with your GitHub account.

	[Git](https://git-scm.com) is version control software, which tracks changes and helps developers work together. Git*Hub* is a very, *very* widely-used, web-based host for Git projects—known as *repositories*. (This is now also owned by Microsoft.)

	You can work with Git [via the command line](https://tutorials.codebar.io/version-control/command-line/tutorial.html) (and even [right from VS Code](https://code.visualstudio.com/docs/sourcecontrol/overview)), but we’ll use their website and GUI *Desktop* app for now.

1.	Head over to [GitHub](http://github.com) (the site) in your browser. [Create a repository](https://docs.github.com/en/get-started/quickstart/create-a-repo) (“repo”) there, named `manuscript`. Make sure it is set to <samp>Public</samp>! We’ll talk about the other options here, but you can ignore them for now.

1. We’ll [*clone* this repo](https://docs.github.com/en/repositories/creating-and-managing-repositories/cloning-a-repository#cloning-a-repository) down to our machines by clicking <samp>Set up in Desktop</samp>. This will link a *local* folder on our computer to the *remote* copy of our code, up on GitHub. (More on this in a bit.)

1. Open the local `manuscript` folder this created in VS Code—either by dragging the folder to app’s icon, or via <samp>File</samp> → <samp>Open Folder…</samp>. You can also open it from GitHub Desktop with <samp>Repository</samp> → <samp>Open in Visual Studio Code</samp>.

	You [can also do this](https://docs.github.com/en/desktop/overview/github-desktop-keyboard-shortcuts) with or <nobr><kbd>⌘</kbd> <kbd>Shift</kbd> <kbd>A</kbd></nobr>.

1. Make an `index.html` file inside this folder. Let’s start with a basic setup:

	```html
	<!doctype html>
	<html>
		<head>
			<title>Hello, world!</title>
		</head>
		<body>
			<h1>Hello, world!</h1>
			<p>This is my <em>first</em> web page.</p>
		</body>
	</html>
	```

	You’ll be using this file for your homework; this is just to get us started. Then let’s open this in the browser!

1. Let’s go through some HTML structure, to start:

	- `<title>`, `<h1>`, `<p>` (note `<em>`)
	- Nested hierarchy, multiple `<article>`, `<blockquote>`, as [block containers](/topic/html/#block-elements)
	- Add [links](/topic/html/#links) and [inline elements](/topic/html/#inline-elements) too

1. And briefly, let’s talk about some *ergonomics*:

	- The [*Command Palette*](https://code.visualstudio.com/docs/getstarted/userinterface#_command-palette): <nobr><kbd>⌘</kbd> <kbd>Shift</kbd> <kbd>P</kbd></nobr>
	- Word wrap: <nobr><kbd>⌥</kbd> <kbd>Z</kbd></nobr>
	- ~~Syntax highlighting and [themes](https://code.visualstudio.com/docs/getstarted/themes)~~
	- ~~[*Tabs* vs. *spaces*](https://www.youtube.com/watch?v=cowtgmZuai0) and render whitespace~~

1. If you hop back over to GitHub Desktop, you’ll notice that our work is now shown there. Use this to review your changes—and if everything is good, write a message, [*commit*](https://docs.github.com/en/desktop/contributing-and-collaborating-using-github-desktop/making-changes-in-a-branch/committing-and-reviewing-changes-to-your-project) them, and push your changes up to GitHub. Think of Git’s *commits* as tracked, commented, and reversible *saves*.

	Keep in mind that *Git* only knows about files that you have committed to your *local* (on your computer) repo. And *GitHub* only knows about files you have pushed to the *remote* (on their computer/server) repo. You have to tell these to sync, with the *push*.

	There are whole methodologies (and many an argument) around Git rules and strategies—but to start, just do what works for you. I’m a fan of many small commits with verbose/specific descriptions (much to other dev’s chagrin). These are like [code comments](/topic/html/#comments)—they are mainly there for *future you*, and later, your colleagues.

1. Check that our `index.html` is now on GitHub by going to the repo there. You can navigate from your profile or jump right from GitHub Desktop with <nobr><kbd>⌘</kbd> <kbd>Shift</kbd> <kbd>G</kbd></nobr>.

1. Now let’s [enable Pages](https://docs.github.com/en/pages/getting-started-with-github-pages/configuring-a-publishing-source-for-your-github-pages-site), which is GitHub’s built-in hosting that we will be using to make our work accessible online. Go to the repo’s <samp>Settings</samp> → <samp>Pages</samp> section, and select `main` as your branch.

1. Back out on the repo page, click the <samp>About</samp> <samp>⚙</samp> and check <samp>Use your GitHub Pages website</samp>. In a minute or two, your page should be live, *on the web* at `https://yourname.github.io/manuscript/`!

	Take a bow, and share your link in <a :href="course.channel.slack">Slack</a>!
	You have made a *web page*! <!-- .intro -->

1. And if we have time—let’s make an update to our file. <!-- .note -->

	You’ll now notice that our changes are now shown in GitHub Desktop as a *diff* (for *diff*erence). Red is a subtraction; green is an addition! I find this easiest to understand in the [*split view*](https://github.blog/2020-11-17-introducing-split-diffs-in-github-desktop/).
